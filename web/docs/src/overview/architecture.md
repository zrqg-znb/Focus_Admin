# 系统架构

Focus Admin 采用前后端分离架构，本文档从**设计理念**出发，讲解系统的架构决策、模块边界和数据流转。

## 设计理念

### 前后端分离

系统采用前后端分离架构，核心考量：

- **独立部署**：前后端可独立发布，互不影响
- **独立迭代**：UI 变更不需要后端配合发版
- **API 优先**：后端只提供 API，支持 Web、移动端、第三方系统等多端接入
- **技术解耦**：前后端技术栈独立演进

### Django Ninja 选型

后端 API 框架选择 Django Ninja 而非 Django REST Framework (DRF)，原因：

| 维度 | Django Ninja | DRF |
|------|--------------|-----|
| 性能 | 高（基于 Starlette） | 中等 |
| 类型安全 | 原生 Pydantic 支持 | 需手动处理 |
| 文档生成 | 自动 OpenAPI 3.0 | 需额外配置 |
| 学习曲线 | 简洁直观 | 较为复杂 |

### 模块化架构

系统按职责划分为四个层级，遵循**低耦合、高内聚**原则：

- **Core**：提供认证、权限、组织架构等基础能力
- **Apps**：实现具体业务逻辑（项目管理、绩效等）
- **Common**：抽象公共组件，提供复用能力
- **Scheduler**：解耦异步任务，支持定时调度

### 模块依赖规则

为保持架构清晰，模块间依赖遵循以下规则：

- **Common** 不依赖任何业务模块，只提供抽象基类
- **Core** 可依赖 Common，提供认证、权限等基础能力
- **Apps** 可依赖 Core（获取用户、权限），但 **Apps 模块间不应直接依赖**
- **Scheduler** 通过 Service 层调用业务逻辑，不直接操作 Model

### 分层设计

每个模块内部采用三层架构：

| 层级 | 文件 | 职责 |
|------|------|------|
| **API 层** | `api.py` | 路由定义、参数验证、调用 Service |
| **Service 层** | `services.py` | 业务逻辑、事务管理、跨模块调用 |
| **Model 层** | `models.py` | 数据模型定义、ORM 操作 |

请求流程：

```
Request → Middleware(认证/权限) → Router → Service → Model → Database
```

## 数据同步架构

系统支持从外部数据源同步数据，采用异步任务 + 日志记录的方式：

```
┌─────────────────┐      ┌─────────────────┐      ┌─────────────────┐
│   APScheduler    │      │   Sync Service  │      │   External API  │
│   定时触发器       │────▶│   同步服务       │────▶│   外部数据源     │
└─────────────────┘      └───────┬─────────┘      └─────────────────┘
                               │
                    ┌─────────┼─────────┐
                    │         │         │
                    ▼         ▼         ▼
              ┌─────────┐ ┌─────────┐ ┌─────────┐
              │ SyncLog │ │ Metric  │ │ Business│
              │ 同步日志  │ │ 指标数据  │ │ 业务数据  │
              └─────────┘ └─────────┘ └─────────┘
                    │         │         │
                    └─────────┼─────────┘
                              │
                              ▼
                    ┌─────────────────┐
                    │     MySQL       │
                    │   数据持久化       │
                    └─────────────────┘
```

**同步类型：**

| 同步类型 | 说明 | 数据具象 |
| --- | --- | --- |
| iteration | 迭代数据 | SR/DR/AR 需求数、状态分布 |
| dts | DTS问题单 | DI值、解决率、等级分布 |
| code_quality | 代码质量 | LOC、函数数、重复率 |
| milestone | 里程碑 | QG点日期、风险项 |
| project | 项目信息 | 项目基础信息 |

## 模块通信机制

后端模块间的通信方式：

| 方式 | 说明 | 示例 |
|------|------|------|
| **ForeignKey 关联** | 数据模型级别的关联 | `Iteration.project → Project` |
| **ManyToMany 关联** | 多对多关系 | `Project.managers → User` |
| **Service 层调用** | 业务逻辑层的跨模块调用 | `ReportService 调用 IterationService` |
